\chapter{Computational Experiments}
\label{sec:experiments}

This chapter presents a comprehensive computational study evaluating the performance of our branch-and-bound algorithm for the bilevel knapsack-scheduling problem. We systematically test the algorithm's efficiency, scalability, and effectiveness through multiple experimental phases: verification against complete enumeration, sensitivity analysis exploring individual parameter scaling, grid-based experiments examining combined parameter effects, and benchmarking against a state-of-the-art bilevel solver.

\section{Experimental Setup}

\subsection{Test Environment}
All experiments were conducted on a single-threaded configuration with the following specifications:
\begin{itemize}
    \item \textbf{Hardware:} AMD Ryzen 5 9600X 6-Core Processor (12 logical processors), 32 GB RAM
    \item \textbf{Software:} Python 3.11, Gurobi~\cite{gurobi} 12.0.3
\end{itemize}

\subsection{Performance Metrics}

For each test instance, we recorded:
\begin{itemize}
    \item \textbf{Runtime:} Wall-clock time in seconds
    \item \textbf{Nodes explored:} Number of branch-and-bound nodes evaluated
    \item \textbf{Pruning rate:} Percentage of nodes at which we pruned, i.e., did not further expand the search tree
    \item \textbf{Solution quality:} Optimal makespan $C_{\max}^*$
    \item \textbf{Status:} root\_optimal (optimal solution found at the root node), completed, timeout (overall time limit exceeded), or leaf timeout (individual leaf node evaluation exceeded time limit)
\end{itemize}

\subsection{Test by Pattern}

To comprehensively evaluate algorithm behavior across diverse problem structures, we developed \texttt{PatternTestSet}, a structured benchmark consisting of 140 instances systematically generated using seven distinct patterns (20 instances per pattern). Each pattern targets specific cost-efficiency characteristics and scheduling challenges:

\textbf{Pattern 1: Uniform Ratios}
\begin{itemize}
    \item Jobs with consistent duration-to-price ratios (1.5--3.0)
    \item Durations: 3--15, prices calculated from ratio
    \item Creates balanced, predictable instances with similar cost-efficiency across all job types
    \item Tests algorithm performance on well-structured instances
\end{itemize}

\textbf{Pattern 2: High Variance}
\begin{itemize}
    \item Alternates between cheap-long jobs (duration 10--20, price 1--3) and expensive-short jobs (duration 2--5, price 8--15)
    \item High variance in duration-to-price ratios
    \item Tests algorithm's ability to handle mixed job portfolios with extreme cost-time tradeoffs
\end{itemize}

\textbf{Pattern 3: Increasing Complexity}
\begin{itemize}
    \item Systematic progression: duration $= 3 + 2j$, price $= 2 + j$ for job type $j$
    \item Jobs become progressively longer and more expensive
    \item Structured, predictable difficulty gradient
\end{itemize}

\textbf{Pattern 4: Random Moderate}
\begin{itemize}
    \item Prices: 5--50, durations: 3--30
    \item Moderate ranges with uniform random distribution
    \item Balanced between structure and randomness
\end{itemize}

\textbf{Pattern 5: Extreme}
\begin{itemize}
    \item Prices: 1--100, durations: 1--50
    \item Wide ranges test algorithm robustness to outliers
    \item Potential for highly unbalanced solutions
\end{itemize}

\textbf{Pattern 6: Strong Correlation}
\begin{itemize}
    \item Strong positive correlation between price and duration
    \item Duration $\approx$ 5 $\times$ price with small random perturbations
    \item Tests algorithm when cost and benefit are closely aligned
    \item Instance generation inspired by~\cite{weninger2025bilevel,pisinger2005hard}
\end{itemize}

\textbf{Pattern 7: Subset Sum}
\begin{itemize}
    \item Prices form subset sum instance structure
    \item Durations independent of prices
    \item Stresses upper-level knapsack decision complexity
    \item Instance generation inspired by~\cite{weninger2025bilevel,pisinger2005hard}
\end{itemize}

\noindent Instance sizes range from 6--12 job types, 3--8 machines, and budgets 15--50. Instances are designated Complex\_001 through Complex\_140.


\subsection{Verification Study: Algorithm Correctness}

To verify correctness and demonstrate computational efficiency, we compared our branch-and-bound results against complete enumeration (the most straightforward and exhaustive approach for problems of this type) on all 140 instances in the PatternTestSet. Enumeration guarantees optimality by evaluating every feasible leader decision, but is prohibitively slow. Even for relatively simple instances, it sometimes exceeds the 1-hour time limit. In contrast, our branch-and-bound algorithm consistently solved the same instances in fractions of seconds, always matching the optimal makespans found by enumeration (when available).

\textbf{Verification results:} Out of 140 instances, branch-and-bound and enumeration produced identical optimal makespans for available cases. Enumeration completed within the time limit for 125 instances (89\%), but timed out (runtime $\geq$ 3,600 seconds) on 15 instances (11\%). Even for these cases, branch-and-bound still found and verified the optimal solution rapidly. No failures or discrepancies were observed. This confirms both the correctness and the dramatic speed advantage of our method over the baseline enumeration approach.

\section{Sensitivity Analysis: Single Parameter Scaling}

Having verified correctness, we investigated how algorithm performance scales as individual problem parameters increase. For each parameter (number of job types, number of machines, budget multiplier), we conducted a systematic sensitivity analysis using a 1-hour time limit per instance. For each parameter value, 5 random instances were generated and solved using both the ceiling and MaxLPT bounds. The parameters were increased until we hit the time limit.


\textbf{Instance generation:} For each parameter setting, test instances were generated using the following procedure:
\begin{itemize}
    \item Job prices: Uniformly random integers in [10, 100]
    \item Job durations: Uniformly random integers in [5, 50]
    \item Budget: $B = \text{budget multiplier} \times$ (average job cost) $\times$ (number of machines)
    \item Baseline: 20 jobs, 10 machines, budget multiplier 2.5, node limit 1 billion, time limit 1 hour per instance, leaf timeout 15 minutes
    \item Number of machines, jobs, and budget multiplier varied as specified
    \item Each instance uses a unique random seed for reproducibility
\end{itemize}

For each instance, the branch-and-bound algorithm was run twice: once with the ceiling bound and once with the MaxLPT bound. To capture the full performance profile, we generated instances until at least one instance used the branch and bound tree instead of being solved at the root node for each parameter setting, allowing us to identify the practical limits of our approach and compare the performance of the two bounds under increasing complexity. Performance metrics were recorded for all runs to analyze trends and differences between the bounds as parameters scaled up.

\subsection{Methodology}

\begin{itemize}
    \item Fix two parameters at baseline values
    \item Systematically increase the third parameter
    \item For each parameter value, generate instances with different random seeds until at least one instance requires branch-and-bound search (not solved at root)
    \item Continue increasing the observed parameter until at least one instance times out (exceeds 3,600 seconds)
    \item Record runtime, nodes explored, pruning rate, and solution quality
\end{itemize}

\paragraph{Column meanings and discussion.}
The sensitivity tables report performance metrics as a single parameter (number of job types, machines, or budget multiplier) increases. 
Columns include average runtime in seconds for the ceiling- and MaxLPT bound, the average number of branch-and-bound nodes explored, the average pruning rate (percentage of nodes at which we pruned), 
and the number of instances that were generated until the problem was not solved at the root node due to the early optimality detection mechanism described in Section~\ref{sec:early-optimality-detection} (since this mechanism works identically for both bounds, the value is shown only once per row).

\section{Grid Sensitivity Analysis}

\subsection{Motivation}

While single-parameter sensitivity analysis revealed individual scaling behavior, real-world instances may vary multiple parameters simultaneously. To understand combined parameter effects and identify practical performance boundaries, we conducted a comprehensive grid-based analysis.

\subsection{Methodology}

We constructed a two-dimensional experimental grid:
\begin{itemize}
    \item \textbf{Job types axis:} 4, 8, 12, 16, 20, 24 (6 values)
    \item \textbf{Machine axis:} 2, 4, 6, 8, 10, 12 (6 values)
    \item \textbf{Budget multipliers:} Fixed at 1.3, 2.5 and 5.0 (separate experiments)
    \item \textbf{Repetitions:} 5 random seeds per configuration
    \item \textbf{Total:} $6 \times 6 \times 3 \times 5 = 540$ instances
    \item \textbf{Time limit:} 1 hour per instance
    \item \textbf{Leaf time limit:} 15 minutes per leaf node evaluation. When this limit is exceeded, we record a leaf timeout status and skip the current instance. In our experiments, some leaf nodes took several hours to evaluate, so this limit prevents excessive runtime while still allowing us to capture the performance of the bounds in challenging cases.
\end{itemize}

Budget scales proportionally with machines: $B = \frac{\sum_{i=1}^{n} p_i}{n} \times m \times \text{multiplier}$

This ensures fair comparison—adding 10\% more machines provides 10\% more budget, maintaining consistent resource availability across machine counts.

\paragraph{Summary of findings.}
For all budget multipliers, 
both bounds solve the vast majority of instances to optimality at the root node. Overall, $84,6\%$ of instances are solved at the root node by both bounds.
The MaxLPT bound generally achieves slightly more completed instances and fewer timeouts compared to the ceiling bound, 
especially for higher multipliers. Leaf timeouts are rare and nearly identical for both bounds and occur only when evaluation of a single leaf node exceeds the time limit, which is more likely in harder instances.
This demonstrates that MaxLPT is at least as robust as the ceiling bound, with a small advantage in harder cases and should generally be preferred for practical use.
The computational overhead of MaxLPT is negligible compared to the ceiling bound, while providing better pruning and fewer timeouts, making it the superior choice for most problems.

\subsection{Runtime and Node Count Comparison: Ceiling vs MaxLPT}

To further quantify the practical differences between the ceiling and MaxLPT bounds, 
we compared the runtime and node counts for all grid sensitivity instances (all budget multipliers) that completed successfully without being proven optimal at the root node.
Overall, there were 61 such instances where both bounds completed, allowing for a direct comparison of their performance.
 The following statistics summarize the observed differences:

\begin{itemize}
    \item \textbf{Average time difference (Ceiling -- MaxLPT):} 71.38 seconds
    \item \textbf{Median time difference:} 0.39 seconds
    \item \textbf{Maximal time difference:} 1057.31 seconds
    \item \textbf{Minimal time difference:} $-$0.43 seconds
    \item \textbf{Average absolute time difference:} 71.42 seconds
    \item \textbf{Average difference when Ceiling is faster:} $-$0.06 seconds
    \item \textbf{Average difference when MaxLPT is faster:} 106.23 seconds
    \item \textbf{Ceiling faster:} 20 times
    \item \textbf{MaxLPT faster:} 41 times
    \item \textbf{Equal times:} 0 times
    \item \textbf{Average node count difference (Ceil -- MaxLPT):} 4,782,454
    \item \textbf{Median node count difference:} 68,985
    \item \textbf{Maximal node count difference:} 54,531,975
    \item \textbf{Minimal node count difference:} 22
    \item \textbf{Ceiling explored fewer nodes:} 0 times
    \item \textbf{MaxLPT explored fewer nodes:} 61 times
    \item \textbf{Equal node counts:} 0 times
\end{itemize}

\paragraph{Interpretation.}
The MaxLPT bound is almost always more efficient in terms of node count, 
exploring fewer nodes in every case. In terms of runtime, MaxLPT is faster in the majority of instances, 
with a much larger average speedup when it is faster. 
The average absolute time difference is dominated by a few hard cases where MaxLPT provides a dramatic speedup. 
When Ceiling is faster, the difference is very small (on average $-$0.06 seconds), 
indicating that the computational overhead of MaxLPT is negligible or even beneficial in practice. 
These results confirm that MaxLPT is the preferred bound for both efficiency and robustness in challenging bilevel knapsack-scheduling problems.

% --- Root-Optimal Case: Runtime Comparison ---
\subsection{Root-Optimal Case: Runtime Comparison}

We also analyzed all instances where both the ceiling and MaxLPT bounds proved optimality at the root node 
(i.e., no branch-and-bound tree was explored).
 This scenario represents the easiest cases for the algorithm, where the initial relaxation is already tight enough to certify optimality without any branching.

The following statistics summarize the runtime differences for all such root-optimal cases (across all budget multipliers):

\begin{itemize}
    \item \textbf{Average time difference (Ceiling -- MaxLPT):} $-0.43$ seconds
    \item \textbf{Median time difference:} $-0.08$ seconds
    \item \textbf{Maximal time difference:} $0.14$ seconds
    \item \textbf{Minimal time difference:} $-7.00$ seconds
    \item \textbf{Average absolute time difference:} $0.43$ seconds
    \item \textbf{Average difference when Ceiling is faster:} $-0.44$ seconds
    \item \textbf{Average difference when MaxLPT is faster:} $0.02$ seconds
    \item \textbf{Ceiling faster:} 447 times
    \item \textbf{MaxLPT faster:} 8 times
    \item \textbf{Equal times:} 2 times
\end{itemize}

\paragraph{Interpretation.}
For root-optimal instances, both bounds are extremely fast, 
with typical runtimes well below a second. 
The ceiling bound is slightly faster on average because the MaxLPT bound precomputes several dynamic programming tables, but the differences are negligible in practice. 
The small negative mean and median indicate that the ceiling bound has a minor edge in these trivial cases, 
but the MaxLPT bound is only marginally slower (and sometimes faster). 
Since no branching occurs, the computational cost is dominated by the computatation of the dynamic programming tables and the solution of the scheduling problem. 







\section{Difficulty Analysis by Parameter Configuration}
To comprehend how the difficulty scales when altering the nuber of job types and machines, we present a scatter plot of the MaxLPT bound runtimes on a log scale, with color indicating the number of machines.

We can see that generally the difficulty scales the same for both bounds. Lower numbers of job types and machines are easier, while higher numbers lead to more timeouts. The MaxLPT bound is more robust, with fewer timeouts at higher parameter values, but the overall trend is similar for both bounds. 
This suggests that the underlying problem difficulty is primarily driven by the combinatorial explosion of possibilities as job types and machines increase, and while MaxLPT provides better pruning, it cannot fully mitigate the exponential growth in complexity.

\section{Comparison with MIBS Solver}

\subsection{MIBS Background}

MIBS (Mixed-Integer Bilevel Solver) is a state-of-the-art open-source solver for bilevel mixed-integer linear programs 
developed by Tahernejad, Ralphs, and DeNegre~\cite{Tahernejad2020}. It implements branch-and-cut techniques with specialized bilevel handling. We selected a subset of our grid instances for comparison to benchmark our specialized algorithm against a general-purpose bilevel solver.

\subsection{Experimental Design}

To provide a comprehensive and fair comparison, we testet the mibs solver on a representative subset of our grid sensitivity instances. We selected instances with varying numbers of machines and job types, as well as different budget multipliers, to capture a range of difficulty levels.
\begin{itemize}
    \item \textbf{Instances:} All combinations of 4, 8, and 12 machines with 4, 8, 16, and 20 job types
    \item \textbf{Budget multipliers:} 1.3, 2.5, and 5.0 (separate experiments for each multiplier)
    \item \textbf{Repetitions:} 5 random seeds per configuration, same as in the grid sensitivity analysis
    \item \textbf{Total:} $3 \text{ (machines)} \times 4 \text{ (job types)} \times 3 \text{ (multipliers)} \times 5 \text{ (seeds)} = 180$ instances
    \item \textbf{Time limit:} 3,600 seconds (1 hour) per instance
    \item \textbf{Comparison metrics:} Runtime, nodes explored, solution quality
\end{itemize}
This design allows us to systematically evaluate both solvers across a broad spectrum of problem sizes and difficulty levels, and to evaluate performance of our approach against a general-purpose bilevel solver.

\subsection{Key Algorithm Insights and Practical Findings}

This subsection combines the main practical findings and algorithmic strengths observed in the computational experiments, highlighting both the performance advantages and critical design choices of the proposed branch-and-bound approach.

\begin{itemize}
    \item \textbf{Specialized wins:} Problem-specific branch-and-bound vastly outperforms general-purpose bilevel solver.
    \item \textbf{Practical advantage:} Our algorithm solves instances in seconds that take MIBS minutes to hours.
    \item \textbf{Scalability:} MIBS struggles with 8+ job types and 8+ machines; our algorithm handles them routinely.
    \item \textbf{Dynamic programming precomputation:} Precomputing all knapsack subproblems once per instance amortizes DP overhead across potentially millions of bound computations. This design choice proved critical for performance.
    \item \textbf{Initialization:} Starting with a high-quality incumbent generated by solving the MaxLPT-problem enables immediate pruning of the entire tree for roughly $84.6\%$ of instances.
    \item \textbf{Machine count sensitivity:} More machines exponentially increases follower scheduling complexity. Beyond 12 machines, performance degrades significantly for instances not being solved at the root node. Even then, evaluating the scheduling problem remains challenging and can take significant time. This limitation is due to the combinatorial explosion of possible schedules as machines increase and cannot be overcome by exact methods without significant algorithmic innovations or heuristics.
    \item \textbf{Node limit dependency:} For hard instances, node counts can exceed 100 million before optimality proof. Since the computational cost of evaluating bounds at each node is constant and very low, we can handle such large node counts reasonably well. Future work could explore more aggressive pruning or heuristic guidance to reduce node counts.
    \item \textbf{Memory requirements:} Dynamic programming tables grow with budget and job types. For exceptionally large budgets compared to job type costs, memory issues may arise.
\end{itemize}


\section{Summary}

This chapter provided an extensive computational evaluation of the proposed branch-and-bound algorithm for the bilevel knapsack-scheduling problem, covering correctness, efficiency, scalability, and comparison to state-of-the-art solvers. The main findings are as follows:

\begin{itemize}
    \item \textbf{Correctness:} The algorithm was verified against complete enumeration on 140 diverse benchmark instances, always matching the optimal makespan and dramatically outperforming enumeration in speed (up to six orders of magnitude faster).
    \item \textbf{Sensitivity analysis:} Systematic scaling of job types, machines, and budget multipliers revealed robust performance across a wide range of parameters. Most instances were solved at the root node, with pruning rates often exceeding 80\%.
    \item \textbf{Grid experiments:} In a comprehensive grid of 540 instances, both the ceiling and MaxLPT bounds solved over 84\% of cases at the root node. MaxLPT consistently achieved more completed solves and fewer timeouts, especially for harder instances.
    \item \textbf{Bound comparison:} MaxLPT explored fewer nodes in every challenging case and was faster in the majority, with negligible overhead in easy (root-optimal) cases. The computational cost of MaxLPT is justified by its superior pruning and robustness.
    \item \textbf{MIBS benchmarking:} On a representative subset of 180 grid instances, the general-purpose MIBS solver solved only 35, while the branch-and-bound algorithm (MaxLPT) solved 179. Median speedup over MIBS was nearly the full timeout limit (3,600s), confirming orders-of-magnitude practical advantage.
    \item \textbf{Scalability limits:} The practical boundary for routine solution is around 26 job types and 12 machines in the case where the budget is proportional to the number of job types and the solution cannot be obtained at the root node, with performance degrading for larger instances due to combinatorial explosion in the follower scheduling problem.
    \item \textbf{Algorithmic insights:} Key design choices—such as dynamic programming precomputation, high-quality incumbent initialization, and efficient upper bounding—were critical for performance and scalability.
\end{itemize}

\textbf{Overall,} the specialized branch-and-bound algorithm, especially with the MaxLPT bound, demonstrates excellent scalability and robustness for realistic problem sizes. It consistently outperforms general-purpose solvers, solves nearly all practical instances in seconds, and provides a reliable tool for challenging bilevel optimization problems.
